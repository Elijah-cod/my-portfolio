The architecture of retrieval: when RAG needs more than vectors.

Semantic similarity is useful, but relevance alone is not the same as understanding. Reliable retrieval systems need explicit choices about relationships, ranking, context, permissions, and what the model is allowed to see.

Retrieval-augmented generation is often introduced as a short pipeline: split documents, create embeddings, search a vector index, and place the best chunks in a prompt. That model is a useful starting point. It is not a complete architecture. As soon as the data contains relationships, permissions, versions, or conflicting sources, retrieval becomes a systems problem rather than a similarity-search feature.

Vector search answers one specific question

An embedding model places text with similar meaning near other text in a vector space. This makes vector search effective when the user and the source describe the same idea with different words. It helps us ask: which passages are semantically close to this query?

That is not the same as asking which information is current, authoritative, connected to a particular account, or safe for the current user. A highly similar chunk may describe an old policy. Another may mention the right product but belong to the wrong workspace. A third may contain an answer without the dependency that explains when the answer applies.

Retrieval quality comes from combining useful signals, not expecting one similarity score to represent the whole meaning of relevance.

Graph context adds relationships to relevance

Graph structures become useful when the relationship between pieces of information matters. A workspace owns projects. A project references decisions. A decision supersedes an earlier decision. A user belongs to a team with a specific access level. These facts are difficult to preserve when every source is reduced to an isolated text chunk.

The graph does not replace semantic search. It constrains and expands it. A practical hybrid flow can use vector search to find candidate passages, identify entities connected to those passages, traverse a small number of approved relationships, then re-rank the combined evidence. Semantic search provides recall. Graph traversal provides structure.

Context assembly should be a first-class layer

Many RAG implementations jump directly from retrieval results to prompt construction. I prefer an explicit context assembly layer between them. That layer can remove duplicates, enforce a token budget, group evidence by source, attach timestamps, and preserve citation identifiers. It also creates a clean place to apply permission checks before any text reaches the model. 

candidates = semantic_search(query, workspace_id)
related = graph.expand(candidates, depth=1)
allowed = permissions.filter(user, candidates + related)
ranked = reranker.score(query, allowed)
context = assembler.build(ranked, token_budget=4200)

This separation makes the pipeline observable. If an answer is poor, we can inspect the candidates, the graph expansion, the permission filter, the rank order, and the final context independently. Without those boundaries, debugging becomes prompt guesswork.

Evaluation must match the retrieval design

End-to-end answer quality is important, but it does not explain where the system failed. Retrieval evaluation should measure whether the relevant source appeared, whether it was ranked early enough, whether necessary relationships were included, and whether forbidden evidence was excluded. Generation evaluation can then test faithfulness and usefulness against the assembled context.

A small, representative test set is more valuable than a large collection of vague prompts. Include ambiguous terms, access boundaries, stale documents, multi-hop questions, and queries where the correct behavior is to say that the evidence is insufficient. Those cases expose architectural weaknesses quickly.

The model is the final consumer, not the whole system

A production RAG feature should behave like product infrastructure. It needs latency budgets, permission boundaries, traceable sources, predictable fallbacks, and a way to improve retrieval without rewriting every prompt. Vector search is a strong component of that system. The architecture becomes dependable when vectors are placed alongside graph relationships, explicit ranking, and a context layer that can be inspected.
